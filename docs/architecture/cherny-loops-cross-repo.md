# Cherny loops across the GuitarAlchemist ecosystem

Boris Cherny's three loops are the operating model behind Anthropic's Claude Code. This document maps how each loop is actually implemented across **ga**, **ix**, **Demerzel**, and **tars** — what's working, what's duplicated, what's missing, and the concrete follow-up PRs that would close the gaps.

Maintained from GA because GA has the deepest cross-repo contracts surface and is where most loops *terminate* (quality baselines, RAG retrieval, chatbot delivery). Should be **mirrored or linked** from each sibling's `CLAUDE.md` — see [§ Cross-repo follow-ups](#cross-repo-follow-ups).

---

## The three loops, restated

| # | Loop | One-line | Cherny example |
|---|---|---|---|
| 1 | **Scheduled / autonomous** | Background routines run on a clock or trigger without per-iteration prompting | `/loop 5m /babysit` rebases PRs and answers review comments |
| 2 | **Institutional memory** | Every mistake or steering moment becomes a checked-in prevention rule the next session reads | `CLAUDE.md` accumulates lessons; drops error rate on a curve |
| 3 | **Execution verification** | Agent generates → runs tests/compiler/browser → reads its own stack traces → fixes → repeats | Closed-loop self-correction without human in the loop |

Cherny argues all three multiply each other: scheduling without memory repeats mistakes; memory without verification accumulates folklore; verification without scheduling needs a human to start it.

## Loop 1 — Scheduled / autonomous

### Pattern primitives in this ecosystem

| Mechanism | Where defined | How triggered |
|---|---|---|
| GitHub Actions on `schedule:` cron | `.github/workflows/*.yml` in each repo | GitHub's cron daemon |
| Claude Code `CronCreate` triggers | Session-scoped, persistent across sessions | Local Claude Code scheduler |
| IXQL `pipelines/*.ixql` | `Demerzel/pipelines/` | `demerzel-driver-triggers.yml` workflow + manual `ixql run` |
| `[skip ci]` snapshot commits | Generated by background poll commits | Local pre-commit hook chains |

### Per-repo inventory

**ga** — 7+ workflows, plus `CronCreate`-scheduled background work:
- `chore(quality): snapshot YYYY-MM-DD [skip ci]` commits land hourly-ish on `main` from local background pollers
- `CronCreate` triggers in flight: `trig_01WdRGSqgxah5PD46wg8u4Qq` (QA Architect Tribunal Phase 1, fires 2026-05-18T09:00Z), `trig_01QUrKEsYLPPW4KNLzKZRE2n` (OPTIC-K SAE Phase 1, fires 2026-05-19T09:00Z)
- Pre-commit hook (`Scripts/install-git-hooks.ps1`) gates every commit with `dotnet format` + build

**ix** — 7 workflows:
- `ga-nightly-quality.yml` — **cross-repo** nightly quality run (ix produces, GA consumes via `state/quality/`)
- `adversarial-qa.yml` — recurring adversarial test gate
- `submodule-auto-update.yml`, `wiki-sync.yml` — maintenance loops
- `claude.yml`, `claude-code-review.yml`, `ci.yml`

**Demerzel** — **18 workflows** (the orchestration hub):
- `demerzel-driver-triggers.yml` — schedules IXQL pipelines
- `demerzel-self-improvement.yml` — Loop 2 mechanism on a schedule
- `cross-model-review.yml` — multi-LLM gate on a schedule
- `demerzel-cross-repo-issues.yml` — cross-repo issue lifecycle
- `streeling-daily.yml` — daily cron
- `submodule-notify.yml` — submodule-change fan-out to consumers
- 15+ `pipelines/*.ixql` files including `algedonic-belief-monitor.ixql`, `conscience-cycle.ixql`, `governance-markov.ixql`, `ml-feedback-loop.ixql`, `qa-architect-cycle.ixql`

**tars** — 2 workflows. Lightest scheduling surface; tars is mostly invoked-on-demand.

**mergerisk** — 1 workflow (just launched). No scheduled loops yet.

### Strengths
- Demerzel is a real orchestration hub with 18 active workflows and ≥15 IXQL pipelines.
- Cross-repo scheduling already exists: `ga-nightly-quality.yml` lives in **ix** but produces artifacts **ga** consumes.
- `CronCreate` adds a personal-machine layer above GitHub Actions: an agent can schedule work that survives session restarts without depending on PR pushes.

### Gaps
- **No central registry**. "What's scheduled to run in the next 7 days?" requires reading 25+ workflow files and an unknown set of `CronCreate` triggers across sessions.
- **No literal `/loop` command** equivalent. Cherny's `/loop 5m /babysit` style — a long-lived in-process routine — has no analog. The closest pattern is `CronCreate` + a skill, but each scheduled task is bespoke.
- **No back-pressure signal** when a loop is wedged. Stale `GaMcpServer` PIDs from 2026-05-06 and 2026-05-08 were holding mmap locks for *six days* unnoticed (surfaced 2026-05-12 during the OPTIC-K corpus rebuild attempt).

## Loop 2 — Institutional memory

### Pattern primitives in this ecosystem

| Mechanism | Where stored | Retrieval |
|---|---|---|
| `CLAUDE.md` at repo root | Each repo | Auto-loaded by Claude Code at session start |
| `AGENTS.md` at repo root | ix, Demerzel | Auto-loaded by Codex / other agents |
| `docs/solutions/` with YAML frontmatter | **ga only** | `ce-learnings-researcher` agent does tag-based retrieval |
| `docs/plans/YYYY-MM-DD-*.md` | **ga only** | Glob + grep; sometimes `/feature` skill |
| User `MEMORY.md` + `feedback_*.md`, `project_*.md`, `reference_*.md` | `~/.claude/projects/<repo>/memory/` | Auto-loaded; structured by type |
| `state/` belief snapshots | **ix** (PDCA cycles, knowledge packages) | `ix-skill` CLI, indexed by date |
| `constitutions/` + `policies/` + `sources/` | **Demerzel** | Policy IDs, schema-validated artifacts |
| `tests/behavioral/*.md` | **Demerzel** (one per persona) | Run as actual NUnit-equivalent tests |
| `v2/agents/*.md` with YAML frontmatter | **tars** | Auto-discovered by tars CLI |
| `docs/audits/` + `docs/articles/` | **Demerzel** | Manual link from cycles |

### Strengths
- **GA's `docs/solutions/`** is the most *retrievable* memory in the ecosystem: YAML frontmatter (`module`, `tags`, `problem_type`), category subdirs, and a dedicated agent (`ce-learnings-researcher`) that searches it before implementation work.
- **Demerzel's `tests/behavioral/`** is the strongest *enforceable* memory: each persona's "remember to do X" is a checked-in test, not a comment.
- **User-level `MEMORY.md`** at `~/.claude/projects/<repo>/memory/` is the cleanest example of Cherny's pattern in this ecosystem — every feedback memory has a `Why:` line and a `How to apply:` line, so future agents can judge edge cases rather than blindly applying rules.

### Gaps
- **Format fragmentation.** Four storage mechanisms across four repos:
  - `docs/solutions/*.md` with YAML frontmatter (ga)
  - `state/*.json` belief snapshots (ix)
  - `policies/*.yaml` + `tests/behavioral/*.md` (Demerzel)
  - `agents/*.md` with YAML frontmatter (tars)

  No agent can search across all four with one call. A learning in ix's `state/` is invisible to GA's `ce-learnings-researcher`.
- **No cross-repo learning fan-out.** When ga learns "Coverlet × ILGPU breaks coverage on GPU kernels" (PR #201 chain, 2026-05-13), there's no mechanism that surfaces that to ix's `state/`, even though ix has GPU code with the same vulnerability.
- **mergerisk has zero institutional memory yet** — green-field opportunity to start with the pattern.

## Loop 3 — Execution verification

### Pattern primitives in this ecosystem

| Layer | Mechanism | Where defined |
|---|---|---|
| **Local pre-commit** | `dotnet format` + build (ga), `cargo clippy -D warnings` (ix), `dotnet build` warnings-as-errors (tars) | `Scripts/install-git-hooks.ps1` (ga); native cargo (ix); `Directory.Build.props` (tars) |
| **PR-time single-LLM review** | `claude-code-review.yml` | ga, ix |
| **PR-time multi-LLM review** | `/octo:review` (8 providers), `ce-*` reviewers, `cross-model-review.yml` | ga (Octopus); Demerzel (the workflow) |
| **PR-time gate workflows** | Build / test / static analysis / security | All repos' `ci.yml` |
| **Adversarial QA** | `adversarial-qa.yml` | ix |
| **Cross-repo quality drift** | `ga-nightly-quality.yml` produces baselines under `state/quality/{embeddings,voicing-analysis,chatbot-qa}/`, `ix-quality-trend` aggregates to `docs/quality/README.md` | ix |
| **Tribunal verdict** | `qa-architect-cycle.ixql` emits `qa-verdict.contract.md`-shaped JSON | Demerzel |
| **Behavioral persona tests** | `tests/behavioral/` per persona | Demerzel |
| **Hexavalent confidence gating** | T/P/U/D/F/C states + thresholds (`≥0.9` autonomous … `<0.3` do not act) | Demerzel |

### Strengths
- **Multi-LLM review is empirically load-bearing.** User feedback memory captures it directly: 9 bugs caught in chatbot migration that local tests missed, 11 more in evolution audits. Strongest evidence in this ecosystem for the verification-as-multiplier claim.
- **Hexavalent logic** (Demerzel) is a richer verification primitive than Cherny describes — "Unknown triggers investigation" and "Contradictory triggers escalation" are first-class states, not just pass/fail.
- **Tonight's PR #201 chain** is a live Cherny verification loop:
  1. CI surfaces `ANTHROPIC_API_KEY missing` at `WebApplicationFactory` boot → fix: `UseSetting("AI:ChatProvider", "ollama")` in `TestWebApplicationFactory`
  2. Fresh CI surfaces `DirectoryNotFoundException : obj/Release/.../compressed/` → fix: include `obj/Release/**/compressed/**` in artifact upload
  3. Fresh CI surfaces `ILGPU.InternalCompilerException` from Coverlet's `RecordHit` injection → fix: `[assembly: ExcludeFromCodeCoverage]` on GPU project
  4. Fresh CI surfaces perf SLO breach (4× under coverage) → fix: `TestCategory!=Performance` filter
  5. Fresh CI surfaces `LazyWithExpirationTests.TimerStartsOnFirstAccess_NotOnConstruction` failing on timing → fix: `TestCategory!=Timing` filter + `[Category("Timing")]` on 3 tests
  6. Fresh CI surfaces stale checked-in TRX files inflating publish-action failure count → fix: `git rm` the 6 stale TRX files

  Six layers peeled in one session, each one identified by the *previous* run's failure mode. Pure Cherny.

### Gaps
- **No shared verification harness.** Each repo's CI is bespoke. When ga discovers "Coverlet instrumentation breaks timing-sensitive tests", there's no mechanism that surfaces the same gotcha to tars's CI (which also runs dotnet test under coverage).
- **`/octo:review` lives only in ga's plugin install.** ix and tars don't have multi-LLM PR review by default; they fall back to `claude-code-review.yml` (single LLM).
- **No closed-loop self-correction yet.** Cherny's pattern: agent reads its own stack trace, auto-fixes, re-runs. The current verification flow still requires *human* in the loop between failure and fix. The infrastructure to close this exists (autopilot mode request in user MEMORY.md captures the desire), but it's not wired.

## Cross-repo glue (the connective tissue)

### JSON-on-disk contracts
Canonical handoff pattern. Sibling repos are clones under the same parent dir. Each contract lives in **ga**'s `docs/contracts/` with a versioned schema:

| Contract | Producer | Consumer | Status |
|---|---|---|---|
| `state/voicings/optick.index` (binary) | ix (`ix-voicings` / `ix-optick`) | ga (RAG retrieval) | stable |
| `state/quality/optick-sae/<date>/optick-sae-artifact.json` | ix (`ix-optick-sae`) | Demerzel (`qa_score_quality_drift`), ga | v0.1.x draft |
| `qa-verdict.contract.md` shape | Demerzel (`pipelines/qa-architect-cycle.ixql`) | ga | v0.1.x draft |

`links.supersedes` is the non-breaking baseline-shift pattern. Contracts only freeze at the explicitly named **Phase 4** milestone.

### Galactic Protocol + capability registry
Demerzel's own equivalent: `governance/demerzel/schemas/capability-registry.json` enumerates what each agent in the ecosystem can do. Locked-field changes need cross-repo coordination.

### Cross-repo workflows
- `ga-nightly-quality.yml` (lives in ix, produces for ga)
- `demerzel-cross-repo-issues.yml` (Demerzel orchestrates issue lifecycle across siblings)
- `demerzel-wiki-cross-repo.yml`
- `submodule-notify.yml`

## Cross-repo follow-ups

Concrete PRs that would consolidate the loops. Each lives in its own repo and can be authored independently — no flag-day required.

1. **All repos** — Add a single line to each `CLAUDE.md`:
   ```markdown
   ## Cherny loops
   See `ga/docs/architecture/cherny-loops-cross-repo.md` for the cross-repo loop architecture.
   ```
   So an agent landing in any sibling has a one-hop path to the consolidated picture.

2. **ix** — Mirror GA's `docs/solutions/` pattern. Today ix uses `state/` for beliefs but has no equivalent of GA's tag-indexed past-fixes corpus. Adopting the YAML-frontmatter shape (`module`, `tags`, `problem_type`) means `ce-learnings-researcher` can be extended to search ix too.

3. **Demerzel** — Add a `pipelines/cherny-loop-audit.ixql` that periodically inventories scheduled work across siblings and writes a consolidated report. Demerzel already has the cross-repo machinery (`demerzel-cross-repo-issues.yml`); this just extends to "what scheduled work exists where."

4. **ga** — Migrate the `state/chatbot-reviews/` and `state/quality/memory-curator/` directories to follow the same naming convention as ix's `state/` (`{date}-{short-description}.{type}.json`). One-time renaming, then both repos read the same shape.

5. **mergerisk** — Adopt the pattern from launch:
   - Drop a starter `CLAUDE.md` with the three-loop preamble + a link here
   - Add `docs/solutions/` directory (empty, with a README that explains the YAML schema)
   - Add the `/octo:review` workflow file early so it accumulates review baselines from PR #1

6. **All repos** — Standardize the user-side `MEMORY.md` location. Today each repo's user memory lives at `~/.claude/projects/<dir-encoded-repo-path>/memory/` which is great per-machine but invisible across team members. A `docs/agent-feedback/` (gitignored body, README explaining the pattern) gives team members an opt-in way to share useful prevention rules.

7. **ga + Demerzel** — Close the Cherny self-correction loop. Today the verification loop still needs a human between "CI surfaces failure" and "agent fixes". A `demerzel-autofix.yml` workflow exists in Demerzel — extending it to read CI failure logs and dispatch a fix PR via the appropriate persona would be a real implementation of Cherny's stage 3 closed loop.

## What this doc is *not*

- Not a freeze. The patterns above are descriptive of 2026-05-13 state; everything changes.
- Not a mandate. Each repo's owners decide which follow-ups apply.
- Not an audit. Loop quality isn't measured here. A future companion doc would track "how often do we re-discover the same gotcha across repos?" as a memory-effectiveness metric.
