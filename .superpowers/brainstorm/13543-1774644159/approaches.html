Blue/Green Binary Swap — 3 Approaches

Click the approach you prefer, or tell me in the terminal

A

Junction Swap (Recommended)

How: Two physical directories (bin/blue/, bin/green/). A Windows junction bin/active → blue|green. Server always runs from bin/active/. Build targets the inactive slot. Swap = move junction.

Pros

  • Instant atomic swap (junction move is ~1ms)
  • Zero file copying during swap
  • Works with any launch method (VS, dotnet run, Aspire)
  • Both slots always exist — rollback is just re-pointing
  • SAC trusts the blue slot while green builds

Cons

  • Needs MSBuild OutputPath override
  • Junction management on Windows needs admin or dev mode
  • IDE (VS) needs config to understand the indirection
B

Shadow Copy + Hot Reload

How: Build normally to bin/Debug/. Pre-build hook copies current bin to bin/shadow/. If SAC blocks, server falls back to shadow copy. Uses .NET's AssemblyLoadContext for runtime swap.

Pros

  • No MSBuild changes — normal build flow
  • Runtime hot-swap without process restart
  • Familiar shadow-copy pattern from IIS

Cons

  • Requires custom AssemblyLoadContext (complex)
  • Shadow copy = full directory copy on every build (~153 DLLs)
  • Hot reload doesn't work for startup code changes
  • SAC blocks happen at load time — can't retry mid-process
C

Publish Slots + Launcher Script

How: dotnet publish to deploy/blue/ or deploy/green/. A PowerShell launcher script manages which slot runs. ga start / ga swap / ga rollback commands.

Pros

  • Published output = trimmed, optimized, self-contained
  • Clean separation from build output
  • CLI commands are intuitive
  • Closest to real blue/green deployment pattern

Cons

  • dotnet publish is slower than dotnet build
  • Published DLLs are ALSO new — SAC blocks them too
  • Separate from IDE debugging flow
  • More disk space (two full published outputs)

💡 Recommendation: A (Junction Swap)

It's the only approach where the inactive slot's binaries age in place — building SAC reputation while the active slot runs. By the time you swap, SAC may already trust them. B and C both produce fresh files that SAC will block just the same.

The key insight: don't copy, don't publish — build directly into the slot and let the files sit there gaining trust.
